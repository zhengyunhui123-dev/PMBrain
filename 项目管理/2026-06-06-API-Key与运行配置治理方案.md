# API Key 与运行配置治理方案

## 目标

PMBrain 的密钥、模型、Dream、导入、同步和向量化配置必须有统一来源，不能要求写入 Windows 全局环境变量，也不能散落在 `.env`、脚本、MCP 配置和数据库里各管一段。

## 配置分层

| 层级 | 放什么 | 文件或位置 | 是否提交 |
| --- | --- | --- | --- |
| 私有文件配置 | API key、默认模型、本地数据库路径 | `C:\Users\zhengyunhui\.gbrain\config.json`，备用镜像 `D:\cursor-claude\PMBrain\.gbrain\config.json` | 不提交 |
| DB 运行配置 | Dream/sync/source/model 路由等可运行时调整的配置 | PMBrain 数据库 config 表，通过 `gbrain config set` 写入 | 不提交 |
| 仓库模板 | 字段说明、空模板、加载器 | `.env`、`load-env.ps1`、项目管理文档 | 可提交无密钥内容 |
| MCP 配置 | MCP server URL 和授权 token | `C:\Users\zhengyunhui\.codebuddy\mcp.json` 或本机私有配置 | 不提交真实 token |

```json

  "database_path": "D:\\cursor-claude\\PMBrain\\.gbrain\\brain.pglite",
  "embedding_model": "zhipu:embedding-3",
  "embedding_dimensions": 1024,
  "expansion_model": "mimo:mimo-v2.5-pro",
  "chat_model": "mimo:mimo-v2.5-pro",
  "mimo_api_key": "...",
  "zhipu_api_key": "...",
  "deepseek_api_key": "..."
}
```

说明：

- MIMO 使用 `mimo_api_key`，运行时映射为 `MIMO_API_KEY`。
- 智谱使用 `zhipu_api_key`，运行时映射为 `ZHIPUAI_API_KEY`。
- DeepSeek 使用 `deepseek_api_key`，运行时映射为 `DEEPSEEK_API_KEY`。
- `openai_api_key` 只保留兼容旧逻辑，不作为 MIMO 的首选配置字段。
- `.env` 和 `load-env.ps1` 不保存真实 key；`load-env.ps1` 只从私有 `config.json` 导出当前进程环境变量，供旧工具兼容。

## Dream / 导入 / 向量化流程

1. 把 Markdown 或通过 `--include-office` 支持的 Word/PDF/Excel/CSV 放入 `D:\Obsidian\Valut\PMGbrain`。
2. 确保该目录是有效 git 仓库，并至少有一次 commit。当前该目录只有 `.git`，没有首个 commit，所以 sync 阶段会提示 `No commits in repo`。
3. 同步导入：

```powershell
bun run src/cli.ts sync --repo D:\Obsidian\Valut\PMGbrain --source default --no-pull
```

4. 向量化：

```powershell
bun run src/cli.ts embed --stale
```

5. Dream：

```powershell
bun run src/cli.ts dream --source default
```

空仓库或无提交时，Dream 的 sync 阶段会失败；这不是 key 问题。
