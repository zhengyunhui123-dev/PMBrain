# 将你的 OpenClaw brain 迁移到 gbrain v0.41.2.0 (greenfield)

v0.41.2.0 lens packs 提供了一个一次性导入器，通过新的摄取大教堂（ingestion cathedral）重新摄取你现有的 OpenClaw brain（`~/git/brain/atoms/`、`concepts/`、`ideas/`）。页面以 `imported_from: markdown-greenfield` frontmatter 标记着陆到 gbrain，因此新的 `extract_atoms` + `synthesize_concepts` 周期阶段会跳过它们（无损导入，保留来源信息）。

## 迁移前准备

1. **将 gbrain 升级到 v0.41.2.0+：**
   ```bash
   gbrain upgrade
   gbrain --version  # 应该是 0.41.2.0 或更高版本
   ```

2. **激活 creator pack**（或者如果你也想在同一个 brain 上使用 investor + engineer lenses，则使用 gbrain-everything）：
   ```bash
   gbrain config set schema_pack gbrain-creator
   # 或者
   gbrain config set schema_pack gbrain-everything
   ```

3. **应用 schema migration v94**（take_domain_assignments 表）：
   ```bash
   gbrain apply-migrations --yes
   ```

## 试运行

始终先使用 `--dry-run` 查看导入器会做什么，而不写入任何内容：

```bash
gbrain capture --source markdown-greenfield \
  --repo ~/git/brain \
  --dry-run \
  --limit 100
```

输出报告：
- `emitted` — 可以干净导入的 atoms/concepts/ideas
- `skipped_no_type` — 没有 `type:` frontmatter 的文件（计为良性跳过；不附加审计）
- `skipped_invalid` — 验证失败的文件（这些会附加到 `~/.gbrain/audit/markdown-greenfield-failures-YYYY-Www.jsonl`）

检查审计 JSONL：
```bash
ls ~/.gbrain/audit/markdown-greenfield-failures-*.jsonl
cat ~/.gbrain/audit/markdown-greenfield-failures-*.jsonl | jq .
```

常见失败原因：
- **空 frontmatter** — 文件有 `---` 但没有字段。不是真正的 brain 页面；安全跳过。
- **格式错误的 YAML** — 在 OpenClaw 中修复文件然后重新运行。
- **缺少必需字段** — 通常意味着原始 OpenClaw skill 输出了部分页面；检查文件是否值得保留。

## 实际导入

当试运行看起来干净时，去掉 `--dry-run` 和 `--limit` 标志：

```bash
gbrain capture --source markdown-greenfield --repo ~/git/brain
```

预计完整的 24K 页面集（atoms + concepts + ideas）需要约 30-60 分钟。导入器：

1. 按确定的字母顺序遍历 `atoms/{YYYY-MM-DD}/*.md`、`concepts/*.md`、`ideas/*.md`，因此部分运行恢复会从上次中断的地方继续。
2. 在每个页面的 frontmatter 上标记 `imported_from: markdown-greenfield` + `imported_at: <ISO timestamp>`，在 `metadata.original_frontmatter` 下逐字保留所有原始字段。
3. 以 `mode: 'migration'`（T2）发出每个 IngestionEvent，这绕过了守护进程的 24 小时 DedupWindow。导入器拥有自己的基于 slug 的持久幂等性。
4. 通过 `put_page` 路由，因此页面以正确的 FK 链和可嵌入性着陆。

## 导入后

验证计数是否匹配：
```bash
gbrain stats
# 应该显示约 24K 个类型为 atom/concept/idea 的新页面
```

下一个 `gbrain dream` 周期将：
- 在**新**转录本上运行 `extract_atoms`（跳过带有 `imported_from` 标记的页面 — 你的历史 atoms 被冻结，不会被重新提取）。
- 在**新** atoms 上运行 `synthesize_concepts`（出于相同原因跳过导入的 concepts）。
- 在导入的页面上运行 `extract_facts` — 导入的 atoms/concepts 中的 facts fences 正常填充 facts 表。

## 停用 OpenClaw 的并行 cron

验证导入后，停用 OpenClaw 的并行 atom pipeline cron 条目。在 `~/git/your-openclaw/workspace/cron.json` 中：

- 移除 `atom-pipeline-coordinator`（每 30 分钟 cron）
- 移除 `atom-backfill-coordinator`（每 10 分钟 cron）

不替换为任何东西 — 当 gbrain-creator（或 gbrain-everything）是活动 pack 时，gbrain 的 autopilot 已经在每个 dream 周期内运行 extract_atoms + synthesize_concepts。

OpenClaw skills 本身缩减为薄封装器：

- `content-atom-extractor` → 调用 `gbrain dream --phase extract_atoms`
- `concept-synthesis` → 调用 `gbrain dream --phase synthesize_concepts`
- `atom-backfill-coordinator` → 删除（backfill 现在是通过每个转录本一次 Haiku 调用中的 Source Quote + lesson enrichment 成为 extract_atoms 的一部分）

## 回滚

导入是完全可逆的：

```bash
# 软删除带有该标记的每个页面（72 小时内可恢复）
gbrain query "imported_from:markdown-greenfield" --type atom --json | \
  jq -r '.[].slug' | xargs -I{} gbrain pages delete {}
# 或者软删除窗口过后硬删除
gbrain pages purge-deleted --older-than 0h
```

你的 OpenClaw 的 `~/git/brain/atoms/` + `concepts/` + `ideas/` 目录不会被导入器触及 — 它们仍然是回滚的来源真相。greenfield 导入器只从它们**读取**。

## 部分失败后重新运行

导入器在页面 slug 级别是幂等的：在相同 `--repo` 上重新运行产生零净新页面（每个页面要么全新着陆，要么匹配现有 slug）。如果你在 OpenClaw 中修复了一些验证失败并想仅重试那些：

```bash
gbrain capture --source markdown-greenfield --repo ~/git/brain
```

已导入的页面保持不变；以前失败的页面获得新的尝试机会；审计 JSONL 按周累积（ISO 周文件轮转）。
