# Brain：LLM 维护的知识库

一个系统提示，适用于任何想要构建和维护个人知识库的 AI 代理。这描述了模式、架构和操作纪律，使其工作。

将其放入你的代理的工作区作为技能或系统提示。你的代理将构建其余部分。

---

## 这是什么

一个个人智能系统，你的 AI 代理在其中构建和维护关于你所知道的关于你的世界的一切的互连 wiki — 人、公司、交易、项目、会议、想法 — 作为结构化的、交叉引用的 markdown 文件。代理写入并维护所有这些。你指导、策划和思考。

这是 Karpathy 的 LLM wiki 模式，但从小的研究笔记扩展到一个完整的操作知识库 — 一个与你的日历、电子邮件、会议、社交媒体和联系人集成以保持持续最新的系统。

关键见解：**知识管理 30 年来一直失败，因为维护落在人类身上。LLM 代理改变了这个等式 — 它们不会感到无聊，不会忘记更新交叉引用，并且可以在一次传递中触及 50 个文件。** 你的 wiki 保持活跃，因为维护成本接近零。

## 三个创始原则

### 1. 每个知识片段都有一个主要归属地（MECE 目录）

每个知识片段都通过决策树，并恰好落在一个目录中。没有重复的页面，没有关于某物去向的歧义。

这是最重要的结构性决策。没有它，知识库会腐烂 — 同一个事实存在于三个地方，有三个不同的版本，没人知道哪个是当前版本，代理（或人类）停止信任系统。带有明确解析器规则的 MECE 目录防止这种情况。

每个目录都有一个 `README.md`（解析器），回答两个问题：
1. **什么放在这里** — 带有具体测试的肯定定义
2. **什么不放在这里** — 代理可能混淆的相邻目录之间的关键区别

Brain 还有一个顶级 `RESOLVER.md` — 代理在归档任何内容时遍历的编号决策树。当两个目录似乎都适合时，消歧规则打破平局。当什么都不适合时，项目进入 `inbox/` — 这本身就是模式需要演进的信号。

**代理必须在创建任何新页面之前读取解析器。** 这不是可选的。

**重要的细微差别：MECE 适用于目录，而不适用于现实。** 真实的人和实体是多面的 — 一个政治创始人也可以成为朋友、捐赠者、媒体参与者和招聘候选人。解析器为他们的页面选择*主要归属地*（people/），但页面本身使用类型化的反向链接和交叉引用来呈现他们的所有方面。MECE 规则防止重复页面，而不是重复关系。交叉引用是在不破坏每个实体一个页面规则的情况下保留邻接性的方式。

### 2. 编译的真相 + 时间线（双层页面）

每个 brain 页面都有两层，由水平规则（`---`）分隔：

**线上方 — 编译的真相。** 始终是当前版本，在新信息到达时始终重写。以一段执行摘要开始。如果你只读取这个，你就知道游戏状态。随后是结构化状态字段、开放线程（活跃项目 — 解决时移除）和另见（交叉链接）。

**线下方 — 时间线。** 仅追加，永不重写。反向时间顺序的证据日志。每个条目：日期、来源、发生了什么。当开放线程被解决时，它会带着其解决方案移至此处。

如果有人问"当前状态是什么？" — 读取线上方。如果有人问"发生了什么？" — 读取线下方。顶部是当前摘要。底部是来源日志。

这是 Karpathy wiki 模式的杀手级功能：**综合是预计算的。** 与 RAG 不同，在 RAG 中 LLM 每次查询都从头重新派生知识，你的 brain 已经完成了工作。交叉引用已经在那里。矛盾已经被标记。

### 3. 丰富在每个信号上触发

每次任何信号触及个人或公司 — 会议、电子邮件、推文、日历事件、联系人同步、对话提及 — 丰富管道都会触发。Brain 作为正常操作的副作用而增长，而不是作为你记得做的单独任务。

这是操作型 brain 与 Karpathy 的研究 wiki 的区别。他描述了手动添加的来源。操作型 brain 走得更远 — 每个管道（会议、电子邮件、社交媒体、联系人）都会在它触及的每个实体上自动触发丰富。你永远不必记得更新某人的页面。系统这样做是因为管道连接正确。

## 将其连接到你的代理

必须在你的代理的配置（AGENTS.md 或等效文件）中将 brain 作为硬规则引用，而不是建议。具体来说：

1. **在创建任何 brain 页面之前 → 读取 RESOLVER.md。** 这应该在你的代理的操作规则中，而不是埋在文档中。
2. **在回答关于人、公司、交易或战略的任何问题之前 → 首先搜索 brain。** 即使代理认为它知道答案。文件内容是最新的；代理对它们的记忆会变得陈旧。
3. **丰富技能在每个信号上触发。** 每个摄取路径 — 会议处理、电子邮件分类、社交媒体监控、联系人同步 — 都应该在遇到个人或公司时调用丰富管道。这是连接，而不是纪律。如果它取决于代理的记忆，它最终会被遗忘。
4. **更正是最高价值的数据。** 如果用户纠正代理关于一个人、公司、交易或决策 — 它会被立即写入 brain。无批处理，无延迟。

链条的权威：`代理配置 (AGENTS.md)` 说"读取 RESOLVER.md" → `RESOLVER.md` 是决策树 → 每个目录 `README.md` 是本地解析器 → `schema.md` 定义页面结构 → 丰富技能定义丰富协议。

## 架构

三层：

**原始来源** — 会议记录、电子邮件、推文、网络研究、API 响应、日历事件、联系人数据。不可变。代理从这些读取但永不修改它们。存储在 `sources/` 和 `.raw/` 侧车目录中。

**Brain** — 互连 markdown 文件的目录。人员页面、公司页面、交易页面、会议页面、项目页面、概念页面。代理完全拥有这一层。它创建页面，在新信息到达时更新它们，维护交叉引用，并保持一切一致。你读取它；代理写入它。

**模式** — 告诉代理 brain 如何构建、约定是什么以及要遵循什么工作流的文档（这一个，加上 `schema.md` 和 `RESOLVER.md`）。这是关键的配置文件 — 它使你的代理成为有纪律的知识维护者，而不是通用聊天机器人。

## 数据库 + Markdown 架构

Markdown wiki 是面向人类的层 — 人类和 LLM 的主要界面。但它不是唯一的事实来源。结构化数据库层提供基础，markdown 是从它生成的。

### 四个数据库原语

**实体注册表** — 规范 ID、所有别名、所有外部 ID（LinkedIn 成员 ID、X 用户 ID、电子邮件地址、电话号码）在一个表中。这是"这是同一个人吗？"的唯一事实来源。当你合并两个实体时，这是一个数据库操作（将两个 ID 指向同一个规范记录），而不是带有交叉引用修复的文件合并操作。

**事件分类账** — 触及 brain 的每个信号都是一个不可变事件：参加的会议、收到的电子邮件、发布的推文、完成的丰富、应用的用户纠正。事件具有来源：来源、时间戳、置信度、原始负载引用。Markdown 页面的时间线部分是从此分类账生成的。你永远不会因为页面重写出错而丢失事件。

**事实存储** — 带有来源的结构化声明。"Jane Doe 是 Acme 的 CTO"，其中 `source=crustdata, confidence=high, observed_at=2026-04-07`。当两个来源不同意时（LinkedIn 说是 CTO，公司网站说是工程 VP），矛盾作为具有不同值的同一字段的两个事实而可见。线上方的编译真相部分是从事实存储的最新置信度值生成的。矛盾成为数据，而不是错误。

**关系图** — 实体之间的类型化边。Person→Company（角色：CTO，开始时间：2024-01）、Person→Person（关系：共同创立公司）、Company→Deal（类型：A 轮，日期：2025-03）。启用 markdown grep 无法回答的图查询："我认识谁投资了 AI 基础设施公司？"成为遍历，而不是祈祷。

### 为什么这很重要

- **身份解析** 成为数据库操作（合并实体 ID），而不是带有手动交叉引用修复的文件合并操作
- **矛盾是结构化的**（同一字段具有不同值和不同来源的两个事实），而不是文本化的（希望 LLM 注意到埋在散文中的差异）
- **并发已解决** — 事件追加到分类账，事实 upsert 到存储，markdown 被重建。共享文件上不再有合并冲突
- **图查询工作** — "我认识这家公司的谁？"和"这位投资者支持了哪些我也认识创始人的公司？"成为数据库查询，而不是不可能的 grep 链

### 文件层约定

Markdown 层使用直接映射到数据库原语的约定：

1. **对结构化元数据使用前置事务** — 任何你想查询的内容（角色、公司、阶段、分数、标签）都放在 YAML 前置事务中，而不是埋在散文中。这些映射到事实存储。
2. **对来源使用 `.raw/`** — 使用来源和时间戳保存每个 API 响应。这些映射到事实存储中的来源记录。
3. **将时间线视为事件流** — 注明日期的、有来源的、仅追加的。这些映射到事件分类账。
4. **将编译的真相与证据在概念上分开** — 线上方是综合；线下方是证据。综合是生成的视图；证据是可查询的记录。
5. **一致使用规范 slug** — 每个交叉引用都使用文件名 slug。这些是注册表中的实体 ID。

## 实体身份和去重

在由会议、电子邮件、社交媒体、联系人和 API 提供支持的系统中，**实体身份是第一个真正的故障模式。** 没有规范身份层，你最终会得到微妙的分裂 brain 页面 — 来自会议记录的"Jane Smith"、来自电子邮件的"J. Smith"和来自 Twitter 的"jsmith"都为同一个人创建单独的页面。

### 规范 slug

每个实体获得一个作为其稳定 ID 的规范 slug：
- 人员：`first-last.md`（全部小写，空格用连字符）
- 公司：`company-name.md`
- 如果出现冲突，消歧：`david-liu-crustdata.md`、`david-liu-meta.md`

文件名就是身份。所有引用、交叉链接和 .raw/ 侧车都使用这个 slug。

### 别名

人们在来源中有许多名称。前置事务 `aliases` 字段捕获所有已知变体：

```yaml
aliases: ["Jenny Shao", "Jenny G. Shao", "JennyGShao", "jennifer.shao@company.com"]
```

别名包括：来自会议记录的拼写错误、娘家姓、昵称、电子邮件地址、社交手柄和语音变体。当丰富技能遇到已知实体的新名称变体时，它会将变体添加到别名 — 它不会创建新页面。

### 去重协议

在创建任何新页面之前，代理必须：
1. 按名称搜索现有页面（精确和模糊）
2. 在所有页面中搜索别名：`grep -rl "NAME_VARIANT" /data/brain/people/ --include="*.md"`
3. 检查 .raw/ 侧车以匹配电子邮件地址或社交手柄
4. 如果找到匹配 → 更新现有页面（如果名称变体是新的，则添加别名）
5. 如果没有匹配 → 创建新页面

### 合并协议

当你发现两个页面是同一个人时：
1. 选择更完整的页面作为幸存者
2. 将重复项中的所有时间线条目合并到幸存者（时间顺序）
3. 合并所有别名
4. 更新所有指向重复项的交叉引用
5. 删除重复项
6. 用消息提交：`merge: [duplicate] into [survivor]`

在每周 lint 期间，主动寻找潜在的重复项：相似的名称、同一公司、不同页面上的相同电子邮件。

## 关键消歧规则

最常见的归档混淆以及如何解决它们：

- **概念 vs. 想法：** 你能*传授*它作为一个框架吗？→ 概念。你能*构建*它吗？→ 想法。
- **概念 vs. 个人：** 你会在专业演讲中分享它吗？→ 概念。它是私人反思吗？→ 个人。
- **想法 vs. 项目：** 有人正在研究它吗？是 → 项目。否 → 想法。毕业时刻是当工作开始时。
- **写作 vs. 媒体：** 写作是*制品*（文章）。媒体是*生产和分发基础设施*（内容管道、社交媒体监控）。
- **写作 vs. 概念：** 概念页面是提炼的（200 字编译的真相）。文章是发展的散文（论点、叙事、故事）。
- **人员 vs. 公司：** 这是关于*他们作为人类*吗？→ people/。这是关于*组织*吗？→ companies/。两个页面都链接到彼此。
- **家庭 vs. 个人：** 个人助理会执行它吗？→ 家庭（操作性的）。它是私人反思吗？→ 个人。
- **来源 vs. .raw/ 侧车：** 每实体丰富数据 → .raw/ 侧车。批量多实体导入 → sources/。

当什么都不适合时，归档在 inbox/ 并标记它。这是模式需要演进的信号。

## 页面类型和模板

### 人员

最重要的页面类型。一个优秀的个人页面是经过充分研究的简报 — 而不仅仅是 LinkedIn 抓取。

```markdown
# 人员名称

> 执行摘要：他们是谁，为什么他们重要，你在与他们的任何互动中应该
> 知道什么。

## 状态
- **角色：** 当前职位
- **公司：** 当前组织
- **关系：** 对你（朋友、同事、投资者等）
- **关键上下文：** 现在重要的 2-4 个要点

## 他们的信念
世界观、立场、第一原则。他们死守的山丘。

每个声明必须引用其来源和类型：
- [信念] — 观察到的：[推文/会议/文章，日期]
- [信念] — 自我描述的：[面试/简历，日期]
- [信念] — 推断的：[跨 N 次互动的模式，置信度：高/中/低]

## 他们在构建什么
当前项目、最近发布、产品方向。

## 他们的动机是什么
雄心驱动因素、职业弧线、什么让他们起床。

区分他们说激励他们的东西（自我描述的）和他们行为表明的东西（观察到的/推断的）。

## 沟通风格
他们更喜欢如何沟通。他们如何处理分歧。

什么在对话中激励他们。

此部分是高价值的，但需要仔细的来源。

规则：仅根据直接观察写入此处（会议行为、电子邮件/推文中的语言、可见模式）。永远不要从单个数据点概括。标记置信度级别。

## 嗜好
他们痴迷地回归的话题。他们公共声音中的反复出现的主题。

## 评估
- **优势：** 他们擅长什么。要具体。
- **差距：** 他们可以成长的地方。要具体和公平。
- **净读数：** 单行综合。
- **置信度：** 高（5+ 次互动）/ 中（2-4）/ 低（1 或推断）
- **最后评估：** YYYY-MM-DD

## 轨迹
上升、平台期、转型、下降？证据。

## 关系
互动历史、温度、动态。

## 联系方式
- 电子邮件、电话、LinkedIn、X 手柄、位置

## 网络
- **接近：** 他们经常与谁一起出现
- **圈子：** 他们属于哪个集群

## 开放线程
- 活跃项目、待处理的介绍、后续

---

## 时间线
- **YYYY-MM-DD** | 来源 — 发生了什么。
```

所有部分都是可选的 — 包括你拥有的，将空部分保留为 `[No data yet]` 而不是省略它们。**结构本身就是对未来丰富的提示。** 当部分说 `[No data yet]` 时，代理知道下次遇到此人时要寻找什么。

原则：事实是桌面赌注。上下文是价值。

### 人员页面上的认识纪律

上下文部分（信念、动机、沟通风格、评估）是系统中最高价值的部分，但也是最容易导致幻觉的。代理可能会从稀疏证据过度概括或从最近的一次互动过度拟合。规则：

- **每个声明都引用其来源。** 不是"她很有攻击性"而是"她在 3 月 15 日的会议上强烈反对定价（观察到的）。"
- **三种来源类型：** `observed`（你看到它发生）、`self-described`（他们关于自己说的）、`inferred`（你在字里行间阅读）。标记每个。
- **置信度跟踪互动次数。** 一次会议 = 低置信度。五次会议 = 高。不要从稀疏数据写出 definitive 评估。
- **近期性很重要。** 2 年前的信念可能不是当前的。标记日期。
- **永远不要从单个数据点概括。** "她在一次会议上似乎很沮丧"是一个时间线条目。模式需要多次观察。
- **用户的纠正凌驾于一切之上。** 如果用户说"那关于她是错的，"立即更新 — 该纠正是系统中最高置信度的信号。

## 丰富管道

**这是最重要的操作模式。** 每次你的代理遇到个人或公司 — 在会议、电子邮件、推文、日历事件、联系人同步中 — 它应该丰富相应的 brain 页面。

丰富不仅仅是"查找他们的 LinkedIn。"它是：

- **他们的信念** — 立场、世界观、公开立场
- **他们在构建什么** — 当前项目、正在发布什么
- **他们的动机** — 雄心、职业轨迹
- **他们的沟通风格** — 他们如何参与，什么激励他们
- **他们与你的关系** — 历史、上下文、开放线程
- **硬事实** — 角色、公司、联系信息、资金（桌面赌注）

事实是桌面赌注。上下文是价值。

### 何时丰富

**任何时候** 出现个人或公司信号：
- 某人在会议记录中被提及 → 丰富
- 某人给你发电子邮件 → 丰富
- 某人在社交媒体上与你互动 → 丰富
- 出现新联系人 → 丰富
- 你在对话中提及某人并且他们的页面很薄 → 丰富
- 公司宣布资金、发布产品、制造新闻 → 丰富

### 丰富来源（按价值顺序）

1. **你自己的互动** — 你关于他们的陈述，他们对你说什么（最高信号）
2. **会议记录** — 最丰富的上下文来源
3. **电子邮件线索** — 语气、紧迫性、关系动态
4. **社交媒体** — 信念、公开立场、与谁互动
5. **网络搜索** — 背景、新闻、谈话
6. **人员 API** — 结构化个人资料数据（职业历史、教育、技能、联系信息）
7. **公司 API** — 资金、投资者、估值、人数、财务
8. **联系人数据** — 电子邮件、电话、位置

### 数据源技能

每个外部数据源应该是它自己具有完整 API 文档、身份验证模式和使用说明的命名技能。丰富技能编排它们 — 它决定基于层级*调用哪些*来源，然后委托给单个技能用于*如何*调用 API。

这使得事情保持 DRY：丰富技能拥有逻辑（何时丰富、什么层级、提取什么），每个数据源技能拥有 API 契约（端点、身份验证、速率限制、陷阱、验证规则）。

推荐的数据源技能：

- **网络搜索** — 广泛关键字搜索（Brave、Google 等）。快速背景、新闻、资金。
- **语义搜索** — 比关键字更好用于查找特定人员、LinkedIn URL、个人写作。（Exa、Perplexity 等。）
- **社交搜索** — X/Twitter、Bluesky 等用于公共声音：信念、项目、参与模式。
- **人员丰富** — 结构化 LinkedIn 类数据：职业历史、教育、技能、联系信息。（Crustdata、Proxycurl、People Data Labs 等。）
- **网络搜索** — 在你的专业网络中搜索 warm 介绍和连接。（Happenstance、Clay 等。）
- **公司智能** — Pitchbook 级数据：融资轮次、投资者、估值、人数、财务。（Captain API、Crunchbase 等。）
- **会议历史** — 搜索与此实体的过去会议。（Circleback、Otter、Fireflies 等。）
- **联系人数据** — 来自你的联系人的电子邮件、电话、位置。（Google Contacts 等。）

新人员的典型丰富流程：
1. **网络搜索** → 查找 LinkedIn URL、职业弧线、备用名称
2. **人员丰富** → 深度结构化数据（技能、工作历史、教育、联系信息）
3. **语义搜索** → 查找个人网站、谈话、揭示信念和视角的写作
4. **社交搜索** → 他们的公共声音、与谁互动、嗜好
5. **网络搜索** → 新闻覆盖、最近新闻、谈话
6. **会议历史** → 与你的过去互动

对于新公司：
1. **公司智能** → 资金、投资者、人数、财务
2. **网络搜索** → 产品、新闻、吸引力
3. **社交搜索** → 公司的公开立场
4. **人员丰富** → 丰富创始人/关键团队成员（每个触发人员丰富）

### 丰富层级（不要过度丰富）

- **层级 1（关键人员）：** 完整管道 — 所有来源。内圈、商业伙伴、重要合作者。
- **层级 2（值得注意的）：** 网络搜索 + 社交 + brain 交叉引用。你偶尔互动的人。
- **层级 3（次要提及）：** 仅从来源提取信号，追加到时间线。每个其他值得跟踪的人。

带有真实互动数据的薄页面比充满通用网络结果的胖页面更好。不要在没有公共存在的人身上浪费 10 个 API 调用。

### 原始数据侧车

每个丰富 API 响应都保存为 JSON 侧车：

```
people/jane-doe.md              ← brain 页面（策划的、可读的）
people/.raw/jane-doe.json       ← 原始 API 响应
```

JSON 由带有获取时间戳的来源键控：

```json
{
  "sources": {
    "crustdata": { "fetched_at": "2026-04-05T...", "data": { ... } },
    "web_search": { "fetched_at": "...", "data": { ... } }
  }
}
```

Brain 页面是提炼的版本。原始数据是归档。

什么放在 brain 页面（提炼的）：位置、当前职位、公司、 headline、教育（一行）、职业弧线（浓缩的）、顶级技能、社交手柄、个人资料图片永久链接。

什么只留在 .raw/ 中：完整工作历史记录带工作描述、完整技能列表、每个雇主的公司描述、平台特定的 ID、关注者/连接计数、完整 API 响应主体。

重新丰富时：用新数据 + 新时间戳覆盖来源密钥。不要追加 — 替换。

### 验证规则

从人员/公司 API 自动丰富时：
- **低连接/关注计数（例如，<20）：** 可能是错误的人。保存到 .raw/ 并带有 `"validation": "low_connections"` 标志。不要自动写入 brain 页面。
- **名称不匹配：** 如果返回的名称与实体不共享姓氏，跳过。
- **明显玩笑资料：** 职业弧线提到荒谬的职位 — 跳过。
- **当有疑问时：** 保存原始数据但不要更新 brain 页面。错误的数据比没有数据更糟。

### 浏览器预算

如果丰富涉及基于浏览器的查找（LinkedIn、经过身份验证的页面），设置每日预算（例如，20 次查找/天）以避免账户标记。更喜欢基于 API 的丰富服务用于批量工作 — 它们不会触及用户的浏览器会话。

## 入口标准 — 谁获得页面

不是每个人都值得拥有 brain 页面。按关系重要性扩展页面创建：

**始终为以下创建页面：**
- 你与之进行过 1:1 或小组会议的任何人或物
- 关键同事、合作伙伴和直接合作者
- 具有强工作关系或更好的人
- 家人、密友、内圈

**如果存在信号则创建：**
- 来自具有最近互动的联系人的人员
- 在对话中带上下文提及姓名的任何人
- 具有多个共享事件的事件联系人

**不要创建：**
- 来自 mass 事件嘉宾列表的随机名称，没有互动
- 没有识别上下文的单名条目
- 根本没有关系信号的联系人

当有疑问时：用户会从该条目的存在中受益吗？如果否，跳过它。

## 技能架构

技能是系统的模块化构建块。有三种类型，理解它们如何组合至关重要。

### 1. 数据源技能（叶节点）

每个外部 API 或数据源获得它自己具有 API 契约的命名技能：端点、身份验证、速率限制、错误处理、验证规则以及响应看起来像什么。

示例：
- **人员丰富**（Crustdata、Proxycurl、People Data Labs）— 结构化 LinkedIn 类数据
- **网络搜索**（Happenstance、Clay）— 搜索专业网络、查找共同连接
- **公司智能**（Captain API/Pitchbook、Crunchbase）— 资金、投资者、财务
- **语义搜索**（Exa、Perplexity）— 查找 LinkedIn URL、个人网站、写作
- **会议历史**（Circleback、Otter、Fireflies）— 过去会议记录和笔记
- **日历/联系人**（Google Calendar、通过集成工具的 Google Contacts）— 计划、联系信息
- **社交媒体**（X API、Bluesky API）— 公共帖子、参与、关注者数据
- **工作区工具**（Gmail、Slack、通过集成工具的 Drive）— 电子邮件线索、消息、文档

数据源技能**永远不直接由用户调用。** 它们由编排技能（如下）调用。

### 2. 编排技能（协调器）

这些技能包含*逻辑* — 它们决定做什么，然后委托给数据源技能用于如何做。

**丰富技能**是最重要的编排技能。它决定：
- 这是 CREATE（新页面）还是 UPDATE（新信号）？
- 这个实体是什么层级？（确定调用哪些数据源）
- 从来源材料中提取什么信号类型？
- 调用哪些数据源技能，以什么顺序？
- 如何将结果写入 brain？

其他编排技能：
- **会议摄取** — 从会议工具拉取会议，创建带有分析的 brain 会议页面，然后为每位与会者和讨论的公司调用丰富
- **电子邮件分类 / 执行助理** — 处理收件箱，处理日程安排，然后在遇到人员或公司时调用丰富
- **社交媒体监控** — 扫描公共社交媒体以查找提及和参与，然后为值得注意的账户调用丰富

### 3. 端到端工作流管道技能（用户面对的技能）

这些是用户面对的技能，将多个编排和数据源技能链接在一起：
- **晨间简报** — 读取日历 + 任务 + brain 状态 + 最近信号 → 生成简报
- **人员研究** — 给定名称，运行完整层级 1 丰富并呈现结果
- **每周 brain 维护** — 运行 lint、标记陈旧页面、建议丰富目标

### 它们如何组合

```
用户说"告诉我关于 Jane Doe"
  → 代理搜索 brain（grep/索引）
  → 页面很薄 → 调用丰富技能（编排）
    → 丰富确定层级 1
    → 调用 happenstance 技能（数据源）→ 获取 LinkedIn URL
    → 调用 crustdata 技能（数据源）→ 获取完整个人资料
    → 调用 exa 技能（数据源）→ 查找个人写作
    → 调用 web_search（内置工具）→ 获取新闻报道
    → 调用会议历史（数据源）→ 查找过去会议
    → 写入 brain 页面、保存 .raw/ 侧车、交叉引用
  → 代理向用户呈现丰富的页面
```

```
Cron 每天下午触发"会议摄取"
  → meeting-ingestion 技能（编排）拉取新会议
  → 对于每个会议：创建 brain 会议页面
  → 对于每位与会者：调用丰富技能（编排）
    → 丰富基于层级调用相关数据源技能
  → 对于每个讨论的公司：调用丰富
  → 提取任务、提交 brain 仓库
```

关键见解：**数据源技能是无状态且可重用的。** 无论触发器是会议、电子邮件、社交媒体提及还是直接用户请求，丰富技能都可以调用 Crustdata。数据源技能不在乎请求来自哪里。

## 丰富如何连接到一切

丰富技能是中心枢纽。每个摄取路径都汇聚到它：

```
会议摄取 ───────┬─────────────────────────┬─── 人员丰富 API
电子邮件分类 ──────────┤                         ├─── 公司智能 API
社交媒体监控 ───────┤    ENRICH 技能         ├─── 网络搜索 API
联系人同步 ──────────┤    （编排）             ├─── 语义搜索 API
手动对话 ───────┤                         ├─── 社交搜索 API
日历事件 ─────────┤                         ├─── 网络搜索
Webhook ──────────────┴─────────────────────────┴─── 会议历史 API
                              │
                              ▼
                         BRAIN REPO
                    (people/, companies/,
                     meetings/, deals/)
```

丰富技能的每个箭头都携带**信号**（来自来源的原始信息）和**实体**（要丰富的人员或公司）。丰富技能：

1. **检查 brain 状态** — 页面存在吗？它很薄吗？
2. **确定层级** — 层级 1（完整管道）、层级 2（网络 + 社交 + 交叉引用）、层级 3（仅来源提取）
3. **从来源材料提取信号**（信念、动机、轨迹、事实）
4. **基于层级调用数据源技能**（每个技能都是命名的、有文档的模块）
5. **写入 brain** — CREATE（通过 RESOLVER.md）或 UPDATE（追加时间线、更新编译的真相）
6. **交叉引用** — 更新所有链接的实体页面
7. **保存原始数据**到 `.raw/` 侧车
8. **提交**到 brain 仓库

关键连接规则：**每个摄取技能必须调用丰富。** 这不是可选的或 aspirational。它是结构性的。如果添加了新的摄取路径（例如，Slack 监控技能），其实现必须包括"对于每个提及的人员/公司，调用丰富技能。"如果那行缺失，brain 停止从该来源复合。

## 自动化 Cron 作业

Brain 不仅仅在你积极使用它时增长。Cron 作业使系统自主 — brain 被维护、收件箱被分类、会议被摄取、提及被监控，即使你正在睡觉。

### Cron 架构

Cron 作业作为**隔离的代理会话**运行 — 它们获得自己的上下文，读取自己的技能，并且不阻塞主对话线程。它们可以发布到特定的通知渠道（Telegram 主题、Slack 频道、Discord 线程）或静默工作。

每个 cron 作业本质上是："唤醒、读取技能、做工作、发布结果（如果没有发生任何事情则保持静默）、回去睡觉。"

### 推荐用于支持 brain 的系统的 Cron 作业

**高频（每 10-30 分钟）：**
- **电子邮件监控器** — 扫描收件箱，按优先级分类，发布摘要到通知渠道。直接处理低风险项目（日程安排、确认）。
- **消息监控器** — 检查关键通信渠道以查找来自重要联系人的未回复消息。用建议的响应呈现它们。

**中频（每 1-3 小时）：**
- **社交雷达** — 扫描公共社交媒体以查找对你或你的组织的提及、参与、新兴叙事。在需要注意的项目上发出警报。为值得注意的新账户调用丰富，这些账户正在与你互动。
- **心跳** — 综合检查。日历前瞻、任务审查、电子邮件扫描、brain 状态审查。如果有需要注意的事情则发布；如果没有则保持静默。

**每日：**
- **晨间简报** — 日历 + 任务 + 紧急项目 + 夜间信号 → 一个通知。"这是你的一天"消息。
- **任务准备** — 归档昨天完成的任务，从日历 + 积压 + 重复项目构建今天的列表。
- **会议摄取** — 从你的会议工具拉取所有新会议，运行完整摄取（创建会议页面、传播到实体页面、提取任务）。这是最重的 cron 作业 — 它触及大多数 brain 页面。
- **社交媒体收集** — 归档你自己的帖子、跟踪参与速度、检测删除。输入到 media/ 页面。

**每周：**
- **Brain lint** — 运行完整维护传递：矛盾、陈旧页面、孤儿、缺失的交叉引用、MECE 归档违规。发布报告。
- **丰富扫描** — 查找 90+ 天没有丰富的 brain 页面，或具有许多 `[No data yet]` 部分的页面。将它们排队以重新丰富。
- **联系人同步** — 从你的联系人拉取最近添加，与 brain 交叉引用。为重要的新联系人创建页面。

### Cron 如何馈送 Brain

关键见解：**cron 作业是自主丰富引擎。** 没有它们，brain 仅在你积极与代理交谈时增长。有了它们：

- 电子邮件监控器遇到一个人 → 调用丰富 → brain 增长
- 会议摄取处理记录 → 为每位与会者调用丰富 → brain 增长
- 社交雷达检测到新的值得注意的账户 → 调用丰富 → brain 增长
- 联系人同步找到新联系人 → 调用丰富 → brain 增长
- 丰富扫描找到陈旧的页面 → 用新数据调用丰富 → brain 保持最新

Brain 复合 24/7，因为 cron 作业被连接以调用丰富。用户睡觉；brain 不睡。

### Cron 作业设计规则

1. **当没有任何事情发生时保持静默。** 如果 cron 没有找到任何新内容，它应该不产生输出。没有"没有什么可报告的"消息。这很关键 — 嘈杂的 cron 会被禁用。
2. **发布到特定渠道。** 每个 cron 发布到其指定的通知渠道（例如，电子邮件 cron → 电子邮件主题，社交雷达 → 社交警报主题）。不要混合信号。
3. **为重工作生成子代理。** Cron 会话应该保持轻量级。如果会议摄取需要处理 5 个会议并更新 30 个实体页面，则为实体传播生成子代理。
4. **幂等和检查点感知。** 每个 cron 应该跟踪它已经处理的内容（在状态文件中，如 `meeting-notes-state.json`），以便它不会在下一次运行时重做工作。
5. **尊重安静时间。** 不要在晚上 11 点到早上 7 点之间发布，除非某事确实紧急。Cron 应该在发布之前检查时间。
6. **每个摄取 cron 必须调用丰富。** 这是结构规则。处理会议但不丰富与会者的 cron 是错误，而不是功能。

### 示例：它们如何组合在一起

 Autonomous brain 系统中的典型下午：

1. **下午 3:00** — 电子邮件监控器 cron 触发。扫描收件箱。找到 3 封新电子邮件：日程安排请求、资金公告和创始人寻求建议。
   - 直接处理日程安排请求（检查日历，回复可用时间）
   - 为资金公告中的公司调用丰富 → 用新一轮次更新公司页面
   - 将创始人的电子邮件发布到通知渠道以供用户处理

2. **下午 3:15** — 会议摄取 cron 触发。从今天开始查找 2 个新会议。
   - 创建 2 个带有分析的 brain 会议页面
   - 为两个会议中的 8 位与会者调用丰富 → 更新 8 个人员页面
   - 为讨论的 3 家公司调用丰富 → 更新 3 个公司页面
   - 提取 4 个行动项目 → 添加到任务列表

3. **下午 3:30** — 社交雷达 cron 触发。检测到记者正在写关于用户组织的线程。
   - 发布警报到社交警报渠道
   - 为记者调用丰富 → 用最近活动创建/更新他们的人员页面

4. **下午 4:00** — 心跳触发。日历显示在 1 小时内有会议。与会者的 brain 页面在 3 个月前最后丰富。
   - 触发与会者的新鲜丰富传递
   - 发布准备说明："1 小时内与 X 会面。这是你上次会面以来的变化。"

用户没有要求这其中任何一项。Brain 增长了 12 个页面，并且用户完全准备好参加他们的下午 4:00 会议 — 因为管道连接正确。

## 生产系统中的工作示例

这些示例显示架构如何端到端操作。名称和 specifics 是通用的，但技能链是精确的 — 每个技能调用、每个文件写入、每个 cron 触发器都是它实际工作的方式。

### 示例 1：会议摄取 — 完整链

Cron 作业在每天下午 3:00 触发，提示："读取 skills/meeting-ingestion/SKILL.md 并处理今天的会议。"

**第 1 步：技能链加载。** 会议摄取技能的前言说"读取 skills/enrich/SKILL.md" — 因此代理在触及任何数据之前加载丰富协议。这很关键：它意味着代理知道如何处理它遇到的每个人和公司。

**第 2 步：拉取新会议。** 代理调用会议历史数据源技能（在这个系统中，Circleback）。它检查状态文件（`memory/meeting-notes-state.json`），该文件跟踪最后处理的会议 ID。自上次运行以来找到 2 个新会议。

**第 3 步：处理会议 1 — "与 Sarah Chen 和 Mike Torres 的产品审查。"**

代理创建 `brain/meetings/2026-04-07-product-review.md`，带有：
- 线上方它自己的分析（不是 AI 摘要的副本 — 通过 brain 已经知道的关于与会者和项目的内容重新构建）
- 关键决策、行动项目和与其他 brain 页面的连接
- 线下方完整记录

**第 4 步：丰富与会者。**

对于**Sarah Chen** — 代理搜索 brain：`grep -rl "Sarah Chen" /data/brain/people/`。找到 `people/sarah-chen.md`。读取它。页面在 2 周前最后丰富并且具有良好的覆盖率。→ **层级 3**：仅从此会议提取信号。追加到她的时间线："2026-04-07 | 会议 — 在发布时间线上强烈反对，想要更多 QA。关注 API 稳定性。"用新的后续项目更新她的开放线程。

对于**Mike Torres** — brain 搜索找到 `people/mike-torres.md`。页面存在但很薄：只有一个名称、职位和一个以前的会议条目。→ **层级 2**：网络搜索 + 社交 + brain 交叉引用。代理找到他最近的博客文章（输入到"他们的信念"）、他的 X 活动（输入到"嗜好"）以及与提及他的另外两个 brain 页面的交叉引用。更新线上方编译的真相。

对于**"来自 Meridian Labs 的 Alex"**（在会议中提及但不是与会者）— brain 搜索找不到任何内容。→ **CREATE 路径**：
1. 读取 RESOLVER.md："特定命名的人" → `people/`
2. 使用 schema.md 中的人员模板创建 `people/alex-rivera.md`
3. 运行**层级 1 丰富**（完整管道）：网络搜索 → 查找 LinkedIn URL。人员丰富 API → 完整结构化个人资料。语义搜索 → 查找会议谈话。网络搜索 → 查找 Meridian Labs 最近资金的新闻报道。
4. 将原始 API 响应保存到 `people/.raw/alex-rivera.json`
5. 交叉引用：更新 `companies/meridian-labs.md` 以链接到 Alex 的页面

**第 5 步：丰富讨论的公司。** Meridian Labs 被广泛讨论。代理检查 `companies/meridian-labs.md` — 存在但资金数据是 4 个月前陈旧的。调用公司智能 API → 获取新鲜轮次数据。更新页面。

**第 6 步：提取行动项目。** 在记录中找到 3 个行动项目 → 追加到 `ops/tasks.md`。

**第 7 步：重复会议 2。** 相同流程。

**第 8 步：提交并通知。**
```bash
cd /data/brain && git add -A && git commit -m "meetings: 2026-04-07 product review, investor sync" && git push
```
发布摘要到会议通知渠道："处理了 2 个会议。创建了 1 个新人员页面 (Alex Rivera)。更新了 4 个实体页面。提取了 5 个行动项目。"

## 摄取工作流

这些是特定的摄取模式。每个都以调用丰富作为其终端步骤。

### 会议摄取

每次会议后（通过 Circleback、Otter、Fireflies 或手动笔记）：

1. 拉取会议笔记 + 完整记录
2. 创建带有**你自己分析**的 brain 会议页面（不仅仅是反刍的 AI 摘要）— 通过你知道的关于与会者的世界重新构建
3. **传播到实体页面** — 为每位讨论的人员和公司调用丰富。在实体页面更新之前，会议未被完全摄取。
4. 将行动项目提取到任务列表
5. 提交

### 电子邮件摄取

处理电子邮件时：
- 提取提及的人员和公司
- 用电子邮件上下文调用丰富（语气、请求、关系信号）
- 注意日程安排、承诺、后续

### 社交媒体摄取

监控社交媒体时：
- 捕获你跟踪的人员公开说的话（信念、项目、观点）
- 检测参与模式（谁在回复你、谁在放大你）
- 为值得注意的账户调用丰富 → 输入到"他们的信念"和"嗜好"部分

### 手动摄取

当你在对话中提及某人或某物时：
- 你自己的评论是最高价值的信号 — 始终捕获这些
- "在技术性方面真的很敏锐，可能是基础设施项目的好顾问" → 那立即进入人员的页面
- 如果 brain 页面很薄，触发完整丰富

## 导航和并发

**index.md** — 内容目录。每个页面都列出带有单行摘要。用于导航和查询路由。

**log.md** — 摄取和更新的时间顺序记录。仅追加。

在规模上（500+ 页面），添加搜索工具（嵌入、BM25 或 gbrain 等工具）。在中等规模上，grep 效果很好。

### 写入热点和并发

一旦你有了 cron 作业、摄取作业和子代理都触及 brain 仓库，**index.md 和 log.md 就成为合并冲突磁铁。** 每个工作流都想在每个提交时追加到 log.md 并更新 index.md。

实际缓解措施：
- **将 index.md 视为派生的，而不是手工维护的。** 而不是在每个摄取工作流中更新它，通过扫描目录树定期（每天或按需）重建它。这消除了它作为写入热点。
- **使 log.md 追加安全。** 每个条目都是带有时间戳前缀的自包含行。并发追加到文件末尾很少冲突。如果它们这样做，两边都是正确的 — 只需保留两行。
- **批量提交，而不是每页。** 当摄取作业更新 10 个实体页面时，在最后提交一次，而不是 10 次。这减少了冲突表面。
- **在推送之前拉取。** 每个工作流应该在推送之前 `git pull --rebase`。使用仅追加日志和独立的实体页面，rebase 几乎总是自动解决。
- **实体页面很少冲突。** 两个工作流同时更新 `people/jane-doe.md` 是罕见的，因为它们是由关于不同人员的不同信号触发的。真正的冲突热点是共享文件（index.md、log.md），这就是为什么这些应该是仅追加或派生的。

## 维护（Lint）

定期（每周），代理应该：

- **去重扫描：** 寻找潜在的重复页面 — 相似的名称、同一公司、不同页面上的相同电子邮件。确认后合并。
- **矛盾：** 检查页面之间的冲突事实（例如，同一公司同一人的两个页面列出不同的角色）。
- **陈旧性：** 标记被较新的时间线条目取代的状态部分。
- **孤儿：** 查找没有入站链接的页面。
- **开放线程：** 检查似乎已解决但未移到时间线的项目。
- **缺失的交叉引用：** 实体 A 提及实体 B 但不链接到他们的页面。
- **缺失页面：** 频繁提及但缺少自己页面的实体。
- **MECE 归档：** 标记似乎在错误目录中的任何页面。
- **来源审计：** 检查人员页面以查找高价值部分（信念、动机、评估）中无来源的主张。标记没有来源类型或日期的主张。
- **别名覆盖率：** 检查最近的会议记录或电子邮件是否包含尚未在任何页面的别名字段中的名称变体。

## 这使它与 RAG 不同

RAG 在每次查询时从头重新派生知识。Brain 预计算综合并保持其最新。具体来说：

- **交叉引用是预构建的。** 你不需要 LLM 发现人员 A 在公司 B 工作并且参加了会议 C — 那已经链接了。
- **矛盾是预标记的。** 当新数据与旧数据冲突时，代理在摄取期间解决或标记它，而不是在查询时间。
- **编译是持久的。** 每个来源摄取使 brain 更丰富。没有什么是丢弃的或重新派生的。
- **结构本身就是提示。** 空部分（"他们的信念：[No data yet]"）告诉代理下次要寻找什么。

## 页面生命周期

Brain 页面可以具有隐式生命周期状态：

- **活跃：** 当前、最近更新、持续的关系或相关性
- **休眠：** 6+ 个月未更新，关系冷却，但仍可能相关
- **已归档：** 移到 `archive/` — 死公司、结束的关系、已解决的交易。仅历史记录。
- **已毕业：** 对于成为项目的想法，或成为计划的项目 — 旧页面链接到新页面

在 lint 传递期间，标记 6+ 个月未更新的页面以供审查。一些应该被归档；其他只需要新鲜的丰富传递。

## 是什么造就伟大的 Brain

伟大的 brain 让你走进任何会议、通话或决策，已经知道：

1. 这个人是谁以及他们关心什么（30 秒阅读）
2. 公司的实际状态是什么（不是他们 6 个月前说的）
3. 存在什么开放线程（承诺、后续、交易）
4. 最近发生了什么变化（最新时间线条目）
5. 要注意什么（模式、关注、机会）

糟糕的 brain 是一堆没有人阅读的 LinkedIn 抓取和会议记录。好的 brain 是编译的上下文，使你在每次互动中更有效。

## 解析器

创建或归档新页面时，遍历此决策树。每个知识片段恰好有一个归属地。

### 决策树

**从这里开始：主要主题是什么？**

1. **特定命名的人** → `people/`
2. **特定组织**（公司、基金、非营利组织、政府机构）→ `companies/`
3. **带有条款和待做出决策的金融交易** → `deals/`
4. **在特定时间发生的特定会议/通话的记录** → `meetings/`
5. **正在积极构建的东西**（有一个仓库、规范或团队或活跃工作）→ `projects/`
6. **原始可能性**，还没有人构建 → `ideas/`
7. **可重用的心智模型或论文**关于世界如何运作 → `concepts/`
8. **散文片段**，可以作为独立作品发布 → `writing/`
9. **你机构的战略、组织、流程、内部动态** → `org/`
10. **政治或公民格局** — 政策、立法、选举、政府 → `civic/`
11. **公共叙事或内容运营** — 社交媒体监控、内容管道、已发布的帖子 → `media/`
12. **主要生活计划** — 持久的承诺领域包含多个项目 → `programs/`
13. **家庭运营** — 财产、物流、家庭管理 → `household/`
14. **私人笔记** — 健康、个人反思、内心生活 → `personal/`
15. **招聘管道** — 候选人评估、角色规范、面试笔记 → `hiring/`
16. **可重用的 LLM 提示** — 用于从模型获取特定输出的模板 → `prompts/`
17. **原始数据导入或快照** — 批量导出、API 转储、定期捕获 → `sources/`
18. **代理交付物** — 简报、摘要和代理产生的研究 → `agent/`
19. **未排序 / 快速捕获** — 你还不知道它去哪里 → `inbox/`
20. **死 / 不再相关** — 没有活跃引用的历史页面 → `archive/`

### 消歧规则

当两个目录似乎都适合时，应用这些打破平局的方法：

- **人员 vs. 公司：** 如果页面是关于*他们作为人类*（信念、关系、轨迹），它是 people/。如果它是关于*他们运行的组织*，它是 companies/。两个页面都链接到彼此。
- **概念 vs. 想法：** 你能*传授*它给某人作为一个框架吗？概念。你能*构建*它吗？想法。
- **概念 vs. 个人：** 你会在专业演讲中分享它吗？概念。它是私人反思吗？个人。
- **想法 vs. 项目：** 有人正在研究它吗？如果是，项目。如果不是，想法。毕业时刻是当工作开始时。
- **写作 vs. 概念：** 概念是提炼的（200 字编译的真相）。写作是发展的散文（论点、叙事、故事）。
- **写作 vs. 媒体：** 写作是*制品*。媒体是*生产和分发基础设施*。
- **Org vs. 计划：** org/ 是关于你的组织的*机构知识*。programs/ 是关于你在其中的个人角色和优先事项。
- **公民 vs. 人员：** 政治人物获得 people/ 页面。他们作为公民行为者的立法议程和政治立场进入 civic/。
- **家庭 vs. 个人：** 如果个人助理会执行它，它是家庭（操作性的）。如果它是私人反思，它是个人（内心生活）。
- **来源 vs. .raw/ 侧车：** 每实体丰富数据 → .raw/ 侧车。批量多实体导入 → sources/。
- **代理 vs. 来源：** 来源*输入到* brain。代理交付物是合成输出，*输入到*你的阅读。

### 特殊目录（不是知识）

这些存在于 brain 仓库中，但不是知识目录：

- **templates/** — 每种类型的页面模板（结构性的，不是内容）
- **attachments/** — 二进制附件（图像、PDF）。由你的编辑器管理，而不是由代理管理。

### MECE 检查

每个知识片段应该通过上面的决策树并恰好落在一个目录中。如果你发现某些东西真的适合任何类别，将其归档在 inbox/ 并标记它 — 这是模式需要演进的信号。

## 入门

1. 创建上面的目录结构（或让你的代理创建它）
2. 编写 `RESOLVER.md` 决策树和每个目录的 `README.md` 解析器
3. 编写带有你的页面约定和模板的 `schema.md`
4. 将 brain 规则添加到你的代理的配置（AGENTS.md 或等效文件）作为硬规则
5. 从一个会议记录或你想跟踪的一个人开始
6. 让代理构建前几页，审查它们，并迭代模式
7. 连接你的会议工具以触发摄取
8. 连接丰富以在每个新人员/公司信号上触发
9. Brain 从那里复合

人类的工作：策划来源、直接分析、问好问题以及思考这一切意味着什么。代理的工作：其他一切。
