## 🐛 错误（2）

### BUG-1：`capture --file`在具有现有frontmatter的文件上加倍frontmatter

**严重性：** 中等
**重现：**
```bash
cat > /tmp/test.md << 'EOF'
---
title: Pre-existing Title
tags: [test, frontmatter]
---

# Pre-existing content

This file already has frontmatter.
EOF

gbrain capture --file /tmp/test.md
gbrain get inbox/2026-05-22-XXXXXX
```

**观察到：** 磁盘上的页面获得两个frontmatter块。外部块具有`title: '---'`（它将内部`---`分隔符解析为标题），并且原始frontmatter按原样保留在body中：

```yaml
---
type: note
title: '---'
captured_at: '2026-05-22T16:06:11.334Z'
captured_via: capture-cli
ingested_via: put_page
ingested_at: '2026-05-22T16:06:13.038Z'
source_kind: put_page
---

---
title: Pre-existing Title
tags: [test, frontmatter]
---

# Pre-existing content
```

**预期：** `buildContent`应该检测现有frontmatter（提交消息说它有一个"看起来像markdown"的启发式方法，用于第一行标题或frontmatter分隔符），并且不要双重包装。内部frontmatter字段应该与capture的字段合并。

**位置：** `src/commands/brainstorm.ts` → `buildContent`函数（与capture共享）。

### BUG-2：POST /ingest在空body上崩溃，出现未处理的TypeError

**严重性：** 中等
**重现：**
```bash
# With valid bearer token:
curl -X POST localhost:3131/ingest \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: text/plain"
```

**观察到：** 返回带有堆栈跟踪的500 HTML错误页面：
```
TypeError: The first argument must be of type string, Buffer, ArrayBuffer,
Array, or Array-like Object. Received undefined
  at serve-http.ts:1508:23
```

**预期：** 路由已经有一个`empty_body`检查（在代码中记录并在E2E套件中测试），但是`/ingest`的body-parser中间件使用`express.raw()`，它返回空POST的`undefined`（没有Content-Length，没有body）。`empty_body`保护在`computeContentHash(body)`调用之后触发，后者在`undefined`上崩溃。

**修复：** 将null/undefined/空buffer检查移到内容哈希计算之前，或者在路由处理程序顶部添加保护：
```typescript
if (!req.body || req.body.length === 0) {
  return res.status(400).json({ error: 'empty_body', message: '...' });
}
```

## ⚠️ 警告（10）

### WARN-1：去重实际上不会去重相同的捕获

**观察到：** 两次捕获相同的文本会产生相同的slug但不同的`content_hash`值：

```
Run 1: slug=inbox/2026-05-22-3d5b671a, hash=1faf3166...
Run 2: slug=inbox/2026-05-22-3d5b671a, hash=f6ee8098...
```

**根本原因：** 内容哈希包括带有frontmatter的完整序列化页面，并且`captured_at`在运行之间更改（它是时间戳）。slug是确定性的（从内容文本派生），因此它正确地映射到同一页面 — 但哈希每次都会更改。

**影响：** 守护程序层中的"24h内容哈希LRU去重"不会捕获重复的捕获，因为哈希不同。`put_page` upsert在slug级别捕获它（覆盖），因此不会创建重复的页面 — 但是基于`content_hash`的去重层对于CLI捕获有效地被绕过。

**建议：** 在添加frontmatter之前从用户输入文本计算content_hash，或者从哈希计算中排除`captured_at`。

---

### WARN-2：具有相同`--type`标志的相同文本会覆盖先前的捕获

对于相同的文本，无论`--type`如何，都会生成相同的slug。第二次使用`--type observation`的捕获会静默地用`--type idea`覆盖第一次。这是正确的行为（slug = 内容哈希），但可能会让用户感到惊讶，他们期望类型更改会产生不同的页面。

---

### WARN-3：`--source`标志因原始外键冲突而崩溃

```
gbrain capture: put_page failed: insert or update on table "pages" violates
foreign key constraint "pages_source_id_fk"
```

错误消息暴露了原始Postgres外键冲突。用户无法知道该怎么做。Capture命令应该捕获此错误并打印人类友好的消息：

```
Error: source 'my-source' is not registered. Register it first:
  gbrain sources add my-source --path /path/to/source
```

### WARN-4：每次捕获后`facts:absorb`连接错误

每次捕获都会记录：
```
[facts:absorb] failed to log gateway_error for inbox/...: No database
connection: connect() has not been called. Fix: Run gbrain init --supabase
```

非致命（exit 0），但很嘈杂。Facts子系统尝试打开CLI捕获路径未初始化的单独连接。

### WARN-5：`capture --help`是最小的 — 不显示标志

```
$ gbrain capture --help
Usage: gbrain capture

gbrain capture - run gbrain --help for the full command list.
```

与具有完整Options部分，示例和成本信息的`brainstorm --help`相比。Capture帮助应该记录`--type`, `--file`, `--source`, `--json`, `--stdin`, `--slug`等。

---

### WARN-6：`capture`, `brainstorm`, `lsd`缺失于`gbrain --help`

这三个命令都没有出现在主帮助文本中。直接调用它们时它们工作，但用户无法从帮助输出中发现它们。`gbrain --help`列出每个其他命令（get, put, search等），但v0.37/v0.38命令缺失。

---

### WARN-7：二进制文件捕获静默成功

```bash
head -c 256 /dev/urandom > /tmp/binary.bin
gbrain capture --file /tmp/binary.bin
```

这成功了，创建了包含二进制垃圾作为内容的页面。应该要么拒绝非文本文件，要么至少警告。

---

### WARN-8：出处列未被`capture`填充

Capture CLI在其JSON输出中报告`"source_kind": "capture-cli"`，但实际数据库行具有：

```json
{ "source_id": "default", "source_uri": null, "source_kind": null }
```

Capture输出中的出处字段不会往返进入持久化页面的`put_page`调用。

---

### WARN-9：Admin register-client忽略scope参数

通过`/admin/api/register-client`注册客户端，使用`"scope": "read write"`创建仅具有`"read"`范围的客户端。Admin端点似乎忽略或默认scope字段。需要手动DB更新以为webhook测试获取`write`范围。

---

### WARN-10：`brainstorm` / `lsd`可能在PgBouncer上挂起或超时

在事务模式PgBouncer环境中，`brainstorm`在成本估算后持续超时，出现`canceling statement due to statement timeout`。混合搜索 + 域库阶段似乎达到了PgBouncer的语句超时。这可能是环境问题，但命令应该优雅地处理超时并报告失败的内容，而不是静默地不产生输出。

## 💡 建议

1. **`capture --help`奇偶校验** — 给它与`brainstorm --help`相同质量的帮助文本。记录每个标志。

2. **主帮助完整性** — 将`capture`, `brainstorm`和`lsd`添加到`gbrain --help`命令列表。这些都是用户面临的功能，无法被发现。

3. **内容哈希稳定性** — 考虑仅从用户输入文本（在frontmatter注入之前）计算去重哈希，以便在守护程序层正确地对相同的捕获进行去重。

4. **出处直写** — `capture`已经知道它是`source_kind: "capture-cli"`。通过`put_page`传递它，以便填充DB列。

5. **二进制文件保护** — 在`capture --file`中拒绝或警告非文本输入。在继续之前检查文件内容或扩展名。

6. **`--source`错误UX** — 捕获FK冲突并打印关于注册源的人类友好提示。
